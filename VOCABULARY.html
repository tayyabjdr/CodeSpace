Concepts

How to use this

Two glossaries. Names of UI parts so you can point at them without describing them. Names of process actions so when I say "I'll rebase that" or "let's merge feat/X" you know what's about to happen.

Naming convention: wherever a UI part has a CSS class in the source, that class is the canonical name. The display name in parentheses is what you'd say in conversation.

Vague

  • "that thing on the left"
  • "the button on the setup screen"
  • "the small dot"
  • "the workspace tab"

Concrete

  • "the workspace item in the sidebar"
  • "the Initialize button on the onboarding form"
  • "the done dot in the sidebar"
  • "the workspace item" (not "tab" — these aren't tabs)

UI states & status indicators

Words to use for visual states.

Focused pane
The pane the user clicked into last (gets a cyan ring). At most one per workspace.
Attended pane
Focused pane + active workspace + window has focus. The done-sound is suppressed for attended panes.
Done pane
A pane whose silence timer fired while unattended. Shows the green tick. Cleared when you focus it.
Pinned pane
A pane the user manually renamed. Auto-namer skips it.
Exited pane
PTY has died (status dot turns grey, .pane.exited).
Draft workspace
A new workspace before Initialize. Sidebar shows ✦ instead of a count. Doesn't persist across reload.
Notifying workspace
An inactive workspace with at least one done pane. Sidebar shows the cyan dot.
Spawned
A workspace whose terminals have been created this session. Set lazily on first activation.

Git actions — basic

When I say these, this is what's about to happen to your repo.

stage"queue this change for the next commit"
Move a file (or part of one) from "modified" to "staged". Doesn't change history. Reversible with unstage.
git add src/renderer/App.jsx
unstage"take it out of the queue"
Move a file back from staged → modified. Your edits are still there.
git restore --staged src/renderer/App.jsx
commit"freeze the staged changes as a snapshot"
Permanently record the staged changes with a message. I never commit without you asking. Each commit gets a hash; nothing is lost — even "deleted" commits are recoverable for a while via git reflog.
git commit -m "feat(sidebar): cyan stripe on active item"
amend"edit the most recent commit"
Replaces the last commit with a new one. I avoid this — I prefer making a new commit instead, because amending after a failed pre-commit hook can lose work.
git commit --amend
restore / discard"throw away uncommitted edits"
Reverts a file to its last-committed state. Destructive — I always ask first.
git restore src/renderer/App.css
stash"shelve uncommitted changes for later"
Pulls all current changes into a temporary stack so you have a clean tree. Get them back with git stash pop.
git stash && git stash pop
status"what's changed in the working tree"
Shows modified, staged, and untracked files. Read-only — runs in seconds, never breaks anything.
git status
diff"show the actual line changes"
Read-only. git diff = unstaged. git diff --staged = staged. git diff master..HEAD = everything on this branch since master.
git diff src/renderer/App.jsx
log"show commit history"
Read-only. git log --oneline for a compact list.
git log --oneline -20

Branches & merging

branch"a parallel line of commits"
In this repo: master is primary; feature work happens on feat/<name>. List with git branch.
switch / checkout"move to a different branch"
Changes the working tree to match the target branch's tip. Uncommitted changes follow you — but if they conflict, switch refuses (you'll need to commit or stash first).
git switch feat/empty-state-button
merge"fold one branch's commits into another"
In this repo's convention, merging feat/X into master creates a single merge: feat/X — <summary> commit (you've been doing this). The commits stay separate but are now reachable from master.
git checkout master && git merge feat/empty-state-button
merge conflict"two branches changed the same lines"
Git pauses and asks you to choose. Files get <<<<<<< markers showing both sides. Resolve, git add, then git commit to finish the merge.
rebase"replay your commits on top of a different base"
Different from merge: rewrites history. Used to make feat/X sit on top of the latest master as if you'd just branched. Don't rebase commits you've already pushed unless you're prepared to force-push.
git rebase master
cherry-pick"copy one commit to this branch"
Takes a commit by hash and replays it here. Good when you want one fix from another branch without merging the whole thing.
git cherry-pick 0893a83
delete branch"remove a branch reference"
After merging, branch references are usually deleted. -d only deletes if merged; -D deletes regardless. I always ask before -D.
git branch -d feat/auto-rename-agents

Remote & PRs

remote"a copy of the repo elsewhere"
In this repo: origin = github.com/tayyabjdr/CodeSpace. Branches that start with remotes/origin/ in git branch -a are pointers to what GitHub had last time you fetched.
fetch"download new commits, don't merge"
Updates your remotes/origin/* pointers but doesn't change your working branches. Safe — never modifies your code.
git fetch origin
pull"fetch + merge in one go"
Updates the current branch with whatever's on its remote counterpart. Can produce merge conflicts if both you and someone else (or the GitHub web UI) made changes.
git pull
push"upload your commits to the remote"
Sends local commits to GitHub so others (and CI / electron-updater) can see them. I always ask before pushing — pushing is visible to others. -u on first push sets up tracking.
git push -u origin feat/empty-state-button
force-push"overwrite the remote with your local history"
Used after rebasing a branch you'd already pushed. Destructive on the remote — I never do this without explicit confirmation, and never on master.
git push --force-with-lease
PR (pull request)"propose merging on GitHub"
A web view of "merge feat/X into master, please". I create these via gh pr create. Doesn't touch your code — just opens the conversation on GitHub.
gh pr create --title "..." --body "..."
tag"a name for a specific commit"
Used for releases — v1.0.0 points permanently at one commit. electron-builder picks up tags for auto-update.
git tag v1.0.0 && git push --tags

Dev & build verbs

dev (run dev server)"hot-reloading dev session"
npm run dev — electron-vite serves the renderer with HMR. Renderer edits hot-reload; main and preload changes need a full restart. Dev runs against an isolated %APPDATA%\CodeSpace-dev\ userData folder, so it can't clobber the installed app's data.
build"produce production bundles"
npm run build — runs Vite for all three targets (main, preload, renderer) and writes out/. No app produced yet, just bundles.
preview"run the production bundles locally"
npm run preview — runs Electron against out/ instead of the dev server. Catches bugs that only show up without HMR / with CSP applied.
package"build an installer"
npm run package — runs electron-builder, produces an NSIS .exe installer. Drops it in dist/.
postinstall"runs automatically after npm install"
Two steps: electron-rebuild -f -w node-pty (recompiles the native PTY against this Electron's V8 ABI), then build-icon.js (re-rasterizes the SVG → ICO).
electron-rebuild"recompile native modules"
node-pty has C++ code that must match Electron's V8. If you upgrade Electron or hit "module was compiled against a different version" errors, this is the fix. Needs Python 3.12 + setuptools on Windows.
test"run the vitest suite once"
npm test — vitest in jsdom. Read-only; safe to run any time.
test:watch"vitest in watch mode"
npm run test:watch — re-runs affected tests on file changes. Long-running; you stop it with Ctrl+C.

Release & auto-update

release"build + publish to GitHub Releases"
npm run release — builds, packages, then runs electron-builder --publish always which uploads the .exe + a latest.yml manifest to a GitHub Release. Once that's live, every running CodeSpace instance will detect it within ~10 s of next launch.
publish"upload the build artifacts"
The "publish" step inside release. Configured in package.json → build.publish (provider: github, owner: tayyabjdr, repo: CodeSpace).
auto-update check"app pings GitHub for newer versions"
Runs 10 seconds after launch. If a newer release is found, electron-updater downloads it in the background; on completion the UpdateToast appears. "Restart now" installs immediately; "Later" defers it to next quit.

App-specific concepts

Words that mean something specific in CodeSpace.

Workspace
A named folder + N agents. Persists. The unit of organization. Not a "tab".
Agent
One terminal pane in a workspace. Shorthand for "a Claude CLI instance running in a PTY". The displayed name is either pinned (name), auto-named (autoName), or default (Agent NN).
PTY
The actual shell process behind a pane. Lives in the main process; survives unmount. One PTY per agent.
PTY pool
The renderer-side fan-out + ring buffer between PTYs and any subscribers (xterm, done-tracker, auto-namer). src/renderer/pty-pool.js.
Ring buffer
Last 64 KB of PTY output kept in memory so a remounted xterm can replay recent context.
Attach / detach
Verbs for connecting an xterm (or any listener) to a PTY in the pool. Mounting a pane attaches; unmounting detaches.
Lazy spawn
Terminals are only created when a workspace is first activated in a session — not on app launch.
Silence timer
The 4 s window the done-tracker uses to decide an agent has stopped responding. Started by an Enter keystroke, reset by every PTY chunk.
Trust prompt
Claude CLI's "Yes, I trust this folder" prompt on first run. pty-manager.js auto-presses Enter for it.
Draft
An unconfigured workspace — name and folder not yet set. Doesn't persist.
Pinned pane
A pane the user manually renamed. Auto-namer skips it forever (until renamed empty).
Done state
An unattended pane after the silence timer fires. Shows the green tick + plays the ding (if volume > 0).
Attended
A pane is attended when it's focused AND in the active workspace AND the window has focus. Done sounds suppressed for attended panes.
Auto-namer
The pipeline that calls claude -p on the PTY tail to produce a 3–5 word title for unpinned panes after every "done" event.
Bridge
The contextBridge surface — window.electronAPI. If it fails to load, the renderer shows the "BridgeMissing" fallback.
CSP
Content Security Policy — applied only in production builds (Vite HMR needs 'unsafe-eval' in dev).
conpty
Windows' modern PTY API. node-pty uses it. Several known quirks (orphan processes, AttachConsole errors) come from here — see CLAUDE.md "Windows quirks".

When in doubt: name the file + the class, e.g. "the .sb-item-dot in Sidebar.jsx".
